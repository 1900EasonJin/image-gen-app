# 启动速度优化计划

## 现状分析

### 当前加载流程（总耗时 ~13秒）

```
用户打开应用
  → Electron 启动 + Express 初始化 (~200ms)
  → 浏览器加载 HTML/CSS/JS (~100ms)
  → 前端 init() 开始
    → sidebarInit()
      → autoConnectSavedProviders()  ← /api/providers/status (~13s !!)
      → refreshModelList()           ← /api/providers (~12ms)
    → loadSessions()                 ← /api/sessions (~27ms)
  → 界面可用
```

### 瓶颈定位

| API | 耗时 | 原因 |
|-----|------|------|
| `/api/providers/status` | **~13s** | 每次启动都调用两个供应商的 connect() 验证 API Key |
| `/api/providers` | ~12ms | 本地读取，无问题 |
| `/api/sessions` | ~27ms | 本地文件读取，无问题 |

`/api/providers/status` 慢的根本原因：
1. **阿里云**：connect() 中发送一个真实的生图请求 `dashscope.aliyuncs.com` 来验证 Key → 网络请求 ~6-7s
2. **火山方舟**：connect() 中发送 `GET Models` 请求 → 网络请求 ~3-5s
3. 两个请求是并行的（已用 Promise.all），但总耗时仍是 ~13s

### 核心问题

**每次启动都要做一次完整的 API Key 验证 + 模型列表拉取，这是完全不必要的。** API Key 只在用户首次输入或手动刷新时才需要验证。

---

## 优化方案：缓存优先 + 后台刷新

### 核心思路

**界面立即展示缓存数据，后台静默刷新最新数据。** 用户感知到的加载时间从 13s → < 200ms。

### 改动清单

#### 1. 后端：新增轻量级状态缓存 API

**文件**: `server/routes/providers.js`

新增 `/api/providers/cached-status` 接口：
- 从 `~/.image-gen-v2/config.json` 读取本地缓存
- 缓存包含：provider 列表、models、maskedKey、连接状态
- 响应时间 < 5ms
- 保留 `/api/providers/status` 作为强制刷新接口

```js
// 缓存文件结构
{
  "providerCache": {
    "volcengine": {
      "connected": true,
      "models": [...],
      "maskedKey": "50b945••••••••2cf8",
      "cachedAt": "2026-06-12T..."
    },
    "aliyun": { ... }
  }
}
```

#### 2. 后端：连接成功时自动更新缓存

**文件**: `lib/crypto-store.js`

在 `saveApiKey()`、`addCustomModel()`、`removeCustomModel()` 时同步更新 `providerCache`。
在 `/api/providers/status` 成功返回时也更新缓存。

#### 3. 后端：阿里云 connect() 优化 — 跳过验证请求

**文件**: `providers/aliyun.js`

当前阿里云的 `connect()` 发送一个真实生图请求来验证 Key，耗时极长。
改为：
- 只做格式校验（Key 非空）
- 返回硬编码的模型列表（阿里云没有模型列表 API）
- 验证推迟到实际生图时自然失败

```js
async connect(apiKey) {
  if (!apiKey || apiKey.trim() === '') {
    return { success: false, error: 'API Key 不能为空' };
  }
  // 直接返回模型列表，不做网络验证
  return {
    success: true,
    provider: { id: 'aliyun', name: '阿里云百炼 (Qwen + Wan)' },
    models: [...]
  };
}
```

#### 4. 前端：启动时先读缓存，后台刷新

**文件**: `public/js/components/sidebar.js`

```
init()
  → fetchCachedStatus()        ← <5ms，立即填充模型列表
  → refreshModelList()         ← <12ms，界面已可用
  → backgroundRefresh()        ← 后台静默刷新
```

#### 5. 前端：所有请求并行 + 非阻塞

**文件**: `public/js/app.js`

```
init()
  → Promise.all([
      sidebarInit(),    // 缓存数据立即加载
      loadSessions(),   // 历史记录并行加载
    ])
  → 界面立即可用
```

## 预期效果

| 阶段 | 优化前 | 优化后 |
|------|--------|--------|
| 缓存加载 | 无 | **< 50ms** |
| 模型列表显示 | 等 ~13s | **< 100ms**（缓存数据） |
| 历史会话 | 等 ~13s | **< 100ms**（并行请求） |
| 界面可操作 | ~13s | **< 200ms** |
| 最新模型刷新 | 同上 | 后台 3-5s（无感） |

---

## 实施顺序

1. **`lib/crypto-store.js`** — 添加缓存读写函数
2. **`providers/aliyun.js`** — 去除验证网络请求
3. **`server/routes/providers.js`** — 添加缓存接口 + 保存缓存逻辑
4. **`public/js/api.js`** — 添加 `fetchCachedStatus()`
5. **`public/js/components/sidebar.js`** — 缓存优先 + 后台刷新
6. **`public/js/app.js`** — 并行初始化

---

## 注意事项

- 缓存失效：用户手动「刷新模型」时清除旧缓存，写入新缓存
- 首次使用：没有缓存时走原有的 status 流程，会慢一次
- API Key 更换：用户输入新 Key 连接时更新缓存
- 自定义模型：增删自定义模型时同步更新缓存
