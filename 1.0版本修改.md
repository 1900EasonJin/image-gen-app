## 问题 1：生成中时隐藏「暂无图片」占位文字

### 现状

生成过程中 loading 遮罩层虽然显示，但下方的 `canvas-placeholder`（「暂无图片」）仍然是可见的，视觉上显得杂乱。

### 根因

`input-area.js` 的 `handleGenerate()` 只控制 `loadingOverlay` 的显示/隐藏，没有联动隐藏 `canvas-placeholder`。

### 修复方案

**修改 `public/js/components/input-area.js`** — `handleGenerate()` 函数：

在设置 `loadingOverlay.classList.remove('hidden')` 的同时，隐藏 placeholder：

```js
// 设置生成中状态
state.generating = true;
generateBtn.disabled = true;
promptInput.disabled = true;

// ★ 隐藏空状态占位
const canvasPlaceholder = document.getElementById('canvasPlaceholder');
if (canvasPlaceholder) canvasPlaceholder.classList.add('hidden');

loadingOverlay.classList.remove('hidden');
```

## 问题 2：生成图片按数量居中布局优化

### 现状

当前图片网格使用 `grid-template-columns: repeat(auto-fill, minmax(280px, 1fr))`，导致：
- 1 张图片：偏左排列
- 2 张图片：并排但可能偏左
- 3 张图片：2+1 不规则
- 4 张图片：勉强 2×2 但受容器宽度影响

### 期望

- **1 张**：居中单张大图
- **2 张**：水平居中并排
- **3 张**：上方 2 张 + 下方居中 1 张（三角形）
- **4 张**：2×2 居中网格

### 修复方案

#### 2.1 修改 `public/css/layout.css`

在 `.canvas-grid` 基础上新增按数量驱动的布局类：

```css
/* ========== 图片网格 — 按数量居中布局 ========== */

/* 通用：覆盖 auto-fill，使用固定列 + justify-content */
.canvas-grid.centered {
  grid-template-columns: none;
  justify-content: center;
  align-content: center;
}

/* 1 张：居中单张大图 */
.canvas-grid.count-1 {
  grid-template-columns: minmax(400px, 600px);
  max-width: 640px;
  margin: 0 auto;
}

.canvas-grid.count-1 .image-card {
  aspect-ratio: auto;
  max-height: 70vh;
}

.canvas-grid.count-1 .image-card img {
  object-fit: contain;
}

/* 2 张：并排 */
.canvas-grid.count-2 {
  grid-template-columns: repeat(2, minmax(300px, 450px));
  justify-content: center;
}

/* 3 张：上 2 + 下 1 */
.canvas-grid.count-3 {
  grid-template-columns: repeat(2, minmax(280px, 380px));
  justify-content: center;
}

/* 将第 3 张居中到第二行 */
.canvas-grid.count-3 .image-card:nth-child(3) {
  grid-column: 1 / -1;
  justify-self: center;
  max-width: 380px;
}

/* 4 张：2×2 */
.canvas-grid.count-4 {
  grid-template-columns: repeat(2, minmax(280px, 420px));
  justify-content: center;
}

/* 5+ 张：自适应网格（原 auto-fill 行为） */
.canvas-grid.count-many {
  grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
}
```

#### 2.2 修改 `public/js/components/result-grid.js`

在 `renderResult()` 方法中，根据图片数量添加对应的 CSS 类：

```js
export function renderResult(images) {
  if (!images || images.length === 0) {
    showEmpty();
    return;
  }

  canvasPlaceholder.classList.add('hidden');
  canvasGrid.classList.remove('hidden');
  canvasGrid.innerHTML = '';

  // ★ 根据数量设置布局类
  const count = images.length;
  canvasGrid.className = 'canvas-grid'; // 重置
  if (count === 1) canvasGrid.classList.add('count-1');
  else if (count === 2) canvasGrid.classList.add('count-2');
  else if (count === 3) canvasGrid.classList.add('count-3');
  else if (count === 4) canvasGrid.classList.add('count-4');
  else canvasGrid.classList.add('count-many');

  // ... 后续渲染卡片逻辑不变
}
```

注意：移除旧代码中 `canvasGrid.classList.add('hidden')` 后的直接 `className` 赋值，改为先清除再按类添加。

## 问题 3：迭代链条缩略图不显示

### 现状

画布下方迭代链条中，每个节点的缩略图显示为破损图片（空白）。

### 根因

**`server/routes/generate.js`** 在存储迭代时，只保存了 `{ id, localPath }`：

```js
images: savedImages.map((img) => ({
  id: img.id,
  localPath: img.localPath,
})),
```

而 **`public/js/components/iteration-chain.js`** 渲染时尝试读取 `dataUrl` 或 `url` 字段：

```js
thumb.src = (iter.images && iter.images[0]?.dataUrl) || iter.images?.[0]?.url || '';
```

两边的字段名不匹配，导致 `src` 始终为空字符串。

### 修复方案

**方法 A：修改前端 `iteration-chain.js`（推荐，改动最小）**

将缩略图 src 改为通过 `/api/images/${id}` 接口获取：

```js
// 旧代码
thumb.src = (iter.images && iter.images[0]?.dataUrl) || iter.images?.[0]?.url || '';

// 新代码
const imgId = iter.images?.[0]?.id;
thumb.src = imgId ? `/api/images/${imgId}` : '';
```

**方法 B：修改后端（备选，数据更完整）**

在 `server/routes/generate.js` 的迭代存储中也写入 `dataUrl`。但这会增加会话文件体积，不推荐。

**推荐采用方法 A**，仅改动一行。

### 变更文件

- `public/js/components/iteration-chain.js`

## 问题 4：修改图片（图生图）返回 400 错误

### 现状

点击图片悬停浮层的 ✏「修改」→ 输入修改描述 → 点击「开始修改」→ 后端返回 400。

### 根因分析

**4.1 火山方舟 (volcengine) 的 `img2img` 问题**

当前 `providers/volcengine.js` 的 `img2img` 方法：

```js
body: JSON.stringify({
  model,
  prompt,
  n: n || 1,
  size: size || '2K',
  response_format: 'b64_json',
  ...(referenceImage && { image: referenceImage }),
}),
```

`referenceImage` 从前端传来的是完整 data URI 格式：`data:image/png;base64,iVBORw0KG...`

但火山方舟 API 的 `image` 字段要求的是：
- **纯 base64 字符串**（不含 `data:image/...;base64,` 前缀），或
- **图片 URL**

直接传入 data URI 会导致 API 无法解析，返回 400。

**4.2 阿里云百炼 (aliyun) 的 `img2img` 问题**

`providers/aliyun.js` 的 `_generateQwen` 方法处理参考图：

```js
if (referenceImage) {
  messages[0].content.unshift({
    image: referenceImage.startsWith('data:')
      ? referenceImage
      : `data:image/png;base64,${referenceImage}`,
  });
}
```

阿里云百炼 Qwen-Image 的多模态 API 要求 `image` 字段是 **公网可访问的 URL**（如 OSS 地址），不支持内联 base64 图片。直接传 base64 data URI 会返回 400。

同样，Wan 系列的 `_generateWan` 也面临类似问题。

### 修复方案

**核心思路：在服务端将参考图统一转为 base64 纯字符串（去掉 data URI 前缀），同时为各 Provider 适配正确的格式。**

#### 4.1 修改 `server/routes/generate.js`

在接收前端请求后，统一预处理 `referenceImage`：

```js
// ★ 统一处理 referenceImage：剥离 data URI 前缀，提取纯 base64
let processedReferenceImage = null;
if (referenceImage) {
  if (referenceImage.startsWith('data:image/')) {
    // data:image/png;base64,xxxx → 提取纯 base64
    const base64Match = referenceImage.match(/^data:image\/\w+;base64,(.+)$/);
    if (base64Match) {
      processedReferenceImage = {
        base64: base64Match[1],
        mimeType: referenceImage.match(/^data:(image\/\w+);/)?.[1] || 'image/png',
        dataUrl: referenceImage,
      };
    }
  } else if (referenceImage.startsWith('http://') || referenceImage.startsWith('https://')) {
    processedReferenceImage = { url: referenceImage };
  } else if (referenceImage.startsWith('/api/images/')) {
    // 本地图片路径，读取文件转 base64
    const imgId = referenceImage.split('/').pop();
    const { loadImage } = await import('../../storage/images.js');
    const img = loadImage(imgId);
    if (img?.dataUrl) {
      const base64Match = img.dataUrl.match(/^data:image\/\w+;base64,(.+)$/);
      if (base64Match) {
        processedReferenceImage = {
          base64: base64Match[1],
          mimeType: 'image/png',
          dataUrl: img.dataUrl,
        };
      }
    }
  }
}
```

然后将处理后的对象传给 `generateImage()`。

#### 4.2 修改 `providers/volcengine.js`

火山方舟 API 的 `image` 字段接受纯 base64 字符串：

```js
async img2img({ model, prompt, referenceImage, n, size, apiKey }) {
  const body = {
    model,
    prompt,
    n: n || 1,
    size: size || '2K',
    response_format: 'b64_json',
  };

  // referenceImage 现在是 { base64, mimeType, dataUrl } 对象
  if (referenceImage) {
    body.image = referenceImage.base64; // ★ 纯 base64，不含前缀
  }

  const resp = await fetch('https://ark.cn-beijing.volces.com/api/v3/images/generations', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${apiKey}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify(body),
  });
  // ... 后续不变
}
```

#### 4.3 修改 `providers/aliyun.js`

##### 4.3.1 Qwen 系列

Qwen-Image 多模态 API 不支持内联 base64，需要先将图片上传到临时可访问的 URL。由于项目没有 OSS 服务，采用以下策略：

- **简化方案（推荐）**：Qwen-Image 的 `img2img` 实际上可以仅通过 prompt 描述（不传参考图也能做修改），或者换用支持 base64 的模型。
- **实际可行方案**：先尝试传 base64，如果 400 则回退为仅 prompt 模式。

但更可靠的方案是：Qwen-Image API 其实在 `messages[].content[].image` 中确实支持 `data:image/...;base64,xxx` 格式。问题可能在于阿里云百炼对图片大小的限制（base64 编码后图片可能太大，超出请求体限制）。

**保守修复**：将图片压缩后再传，或者先转为更小的格式。先试试直接传 data URL（前缀保留），看是否是大小问题。

```js
async _generateQwen({ model, prompt, n, size, apiKey, referenceImage }) {
  const messages = [{ role: 'user', content: [{ text: prompt }] }];

  if (referenceImage) {
    // referenceImage 是 { base64, mimeType, dataUrl }
    // Qwen 多模态 API 接受 data: URL 格式
    messages[0].content.unshift({
      image: referenceImage.dataUrl || `data:${referenceImage.mimeType || 'image/png'};base64,${referenceImage.base64}`,
    });
  }
  // ... 后续不变
}
```

##### 4.3.2 Wan 系列

Wan 通过阿里云百炼调用，`ref_img` 字段也接受 base64（可能也需要纯 base64 或者 data URL）。同样修复格式。

#### 4.4 修改 `providers/index.js`

更新 `generateImage` 函数签名以支持结构化的 `referenceImage` 对象：

```js
export async function generateImage({ provider: providerId, model, prompt, n, size, referenceImage, sessionId, apiKey }) {
  // ...
  const params = { model, prompt, n, size, apiKey: resolvedApiKey };

  if (referenceImage) {
    params.referenceImage = referenceImage; // 传递结构化对象
    return provider.img2img(params);
  }

  return provider.text2img(params);
}
```

## 问题 5：移除「绘图/编辑」模式切换按钮，改为静态文字

### 现状

顶部工具栏中间显示 `[绘图] [编辑]` 两个可点击的 Tab 切换按钮。由于悬停点「修改」已经有完整的图生图交互（弹窗→自动进入编辑模式），顶部的 Tab 切换显得冗余。

### 期望

只显示静态文字「绘图」，不显示按钮。

### 修复方案

#### 5.1 修改 `public/index.html`

将 `<div class="mode-tabs">` 替换为静态标签：

```html
<!-- 旧代码 -->
<div class="mode-tabs" id="modeTabs">
  <button class="mode-tab active" data-mode="draw" id="tabDraw">绘图</button>
  <button class="mode-tab" data-mode="edit" id="tabEdit">编辑</button>
</div>

<!-- 新代码 -->
<span class="mode-label">绘图</span>
```

#### 5.2 修改 `public/css/layout.css`

- 删除 `.mode-tabs`、`.mode-tab`、`.mode-tab:hover`、`.mode-tab.active` 样式
- 新增 `.mode-label` 静态文字样式：

```css
.mode-label {
  font-size: 13px;
  color: var(--text-tertiary);
  font-weight: 500;
  user-select: none;
}
```

#### 5.3 修改 `public/js/components/work-area.js`

移除 Tab 切换相关逻辑，简化为纯初始化：

```js
import state from '../state.js';
import { $ } from '../utils/dom.js';

const statusTag = $('#statusTag');

export function init() {
  // 监听生成状态变化
  window.addEventListener('imagesGenerated', () => {
    if (statusTag) {
      statusTag.textContent = '已生成';
      statusTag.style.color = 'var(--success)';
    }
  });

  window.addEventListener('stateChange', (e) => {
    if (!statusTag) return;
    if (e.detail.generating === true) {
      statusTag.textContent = '生成中...';
      statusTag.style.color = 'var(--accent)';
    } else if (e.detail.generating === false && !state.currentSession) {
      statusTag.textContent = '暂无数据';
      statusTag.style.color = '';
    }
  });
}
```

#### 5.4 修改 `public/js/components/input-area.js`
